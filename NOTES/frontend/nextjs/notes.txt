Note: after 60 secs, the next user request will trigger a background re-generation of the page and then serve the fresh static page to next subsequent user.

revalidateTag, revalidatePath lets you manually refresh cached data/pages instantly instead of waiting for ISR timers.
  await fetch('/posts', { next: { tags: ['posts'] } })
  // Revalidate multiple components depending on same state (Home, Blog) 
  import { revalidateTag } from 'next/cache';
  revalidateTag('posts');
  // Refresh cache for a specific URL route /posts
  import { revalidatePath } from 'next/cache';
  revalidatePath('/posts');

ISG revalidate cache: 
1) Time-based revalidation (Background):
    1.1) Entire component

2) On-demand revalidation (Manual):
    2.1) Path-based: 
          revalidatePath('/posts')
    2.2) Tag-based: 
          fetch('/posts', { next: {tags: ['posts']} });
          revalidateTag('posts');

Next.js caching v/s Tanstack Query caching 

Use Next.js caching when:
 - Data is static or semi-static (Blog posts, product description)
 - SEO is critical 
 - Reduction in client side bundle is needed 
 - Data changes only on hard navigation or server action 

Use Tanstack Query when:
 - UI is highly interactive (Dashboard, real time feeds, chat)
 - Need background sync (refetch data when user switches tabs back to your app)
 - Need optimistic update 

Login auth workflow:

1. Login successful: API sends response access and refresh token in cookies.
    Set-Cookie: accessToken=ACCESS_TOKEN; HttpOnly; Secure; SameSite=Lax
    Set-Cookie: refreshToken=REFRESH_TOKEN; HttpOnly; Secure; SameSite=Lax
2. Request → cookies sent (access + refresh)
3. Access token expired
4. Backend uses refresh token
5. Generates new access token
6. Sends response + Set-Cookie
    Set-Cookie: accessToken=NEW_TOKEN; HttpOnly; Secure; SameSite=Lax
7. Browser updates cookie automatically
8. Next request uses new token

Race Conditions: If the Refresh Token is one-time use (Rotation), concurrent requests might cause a "Used Token" error. Architects solve this by adding a small grace period (e.g., 10 seconds) where an old Refresh Token is still accepted after a new one is issued.

The workflow you described hits a snag when using React Server Components (RSC).

In a Server Component, the fetch happens Server-to-Server (Next.js Server → Backend API).
The Issue: When your Backend sends a Set-Cookie response to the Next.js Server, the browser doesn't see it yet.
The Solution: You must manually proxy that Set-Cookie header from the API response back to the client using the Next.js cookies() header or by forwarding headers in Middleware.
