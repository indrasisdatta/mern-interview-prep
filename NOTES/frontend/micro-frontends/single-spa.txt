Iframes are notoriously difficult for:
    Deep Linking: If a user bookmarks a page inside the Angular app, how does the React 'Shell' know to load that specific sub-route on a refresh?

    Responsiveness: Iframes don't grow with their content easily. If the Angular tool has a long table, the user ends up with 'Double Scrollbars' (one for the frame, one for the browser). This looks 'un-professional' for a Board Meeting demo.

    Communication: You mentioned an Event Bus/RxJS. How do you handle Authentication synchronization? If the user logs out of the React Shell, how do you ensure the Angular Iframe immediately clears its session without a reload?"


The Registry/Single-SPA approach
 - Imagine a "Shell" app that doesn't use Iframes. Instead, it acts as a Router that physically mounts and unmounts entire applications into a single <div>.
 - When the URL is /dashboard: The Shell "Mounts" the React app.
 - When the URL is /admin: The Shell "Unmounts" the React app (clearing its memory) and "Mounts" the Angular app into that same DOM node.

Use single-spa - register applications based on URL.

https://single-spa.js.org/docs/configuration

    // single-spa-config.js
    import { registerApplication, start } from "single-spa";
    // Simple usage
    registerApplication(
        "app2",
        () => import("src/app2/main.js"),
        (location) => location.pathname.startsWith("/app2"),
        { some: "value" },
    );

    // Config with more expressive API
    registerApplication({
        name: "app1",
        app: () => import("src/app1/main.js"),
        activeWhen: "/app1",
        customProps: {
            some: "value",
        },
    });
    start();

    - LIFECYCLES: Each child app must export:
        1. bootstrap(): One-time setup.
        2. mount(): Fires when the route matches (renders to DOM).
        3. unmount(): Fires when leaving the route (cleans up memory/CSS).

If you 'Unmount' the Angular app every time the user clicks back to the React Dashboard, won't that make the 'Switching' feel slow? We'd have to re-initialize the entire Angular framework every time. How do you make that 'Instant' (< 500ms)?
 - Resource caching 
 - Network caching - prefetch Angular bundle in the background while the React dashboard is being accessed. 
    <link rel="modulepreload" /> or service worker to cache Angular vendor files locally 

To prevent CSS conflicts, use shadow DOM as it blocks external CSS. Angular/bootstrap styles must be injected within the shadow root and not document <head>

Create shadow host in shell:
    // In your Shell/Orchestrator
    const appContainer = document.getElementById('angular-app-container');
    const shadowRoot = appContainer.attachShadow({ mode: 'open' });

    // We create a "slot" where the Angular app will live
    const innerContainer = document.createElement('div');
    innerContainer.id = 'angular-root-wrapper';
    shadowRoot.appendChild(innerContainer);

Use Webpack/Vite style-loader:
    {
        loader: 'style-loader',
        options: {
            insert: function(element) {
                // Instead of document.head, we find our Shadow Root
                const host = document.querySelector('#angular-app-container');
                host.shadowRoot.appendChild(element);
            }
        }
    }

For Modals/Tooltips: If the Angular app uses document.body for its portals/modals (like some older versions of ng-bootstrap), the modal will render outside the Shadow DOM and lose its styles. We'd need to configure the Angular 'Portal' strategy to target our Shadow Root instead.


MFE GOVERNANCE: PREVENTING RUNTIME BREAKS (SHARED TYPES & CONTRACTS)

1. THE "CONTRACT" PROBLEM
In Micro-frontends (MFE), teams are decoupled. If the Angular team changes 
a data structure (e.g., API response keys) that the React Shell relies on, 
the app will crash at runtime because there is no shared compiler.

- GOAL: Move from "Trust" to "Automated Verification."

2. THE 3-LAYER GOVERNANCE STRATEGY (STAR-Arch)

    [S] LAYER 1: SHARED TYPES LIBRARY (The Source of Truth)
        - Create a "Types-Only" NPM package or Git Submodule (e.g., @org/api-types).
        - Contains ONLY TypeScript Interfaces and Enums.
        - Implementation: Both the React and Angular teams must import this. 
        If a team changes an interface, the other team's CI/CD build 
        fails immediately during the 'type-check' phase.

    [T] LAYER 2: E2E SMOKE TESTS (The Integration Guardrail)
        - Unit tests only check an app in isolation. You need "Cross-App" tests.
        - Implementation: Use Playwright or Cypress in the CI/CD pipeline.
        - Scenario: Every time the Angular team pushes code, the CI loads the 
        Production Shell + the New Angular Bundle and clicks through the 
        "Global Navigation." If the Shell breaks, the deployment is blocked.

    [A] LAYER 3: PR TEMPLATES & IMPACT LABELS (Manual Governance)
        - Force developers to acknowledge potential breaks before merging.
        - Tool: GitHub/GitLab PR Templates.
        - Checklist: 
        [ ] "Does this change a shared utility or type?"
        [ ] "Have I verified the CSS doesn't leak into the Shell?"
        [ ] "Is the 'Contract' between the Apps still valid?"

3. ARCHITECT'S RECOMMENDATION: "DECOUPLED BUT VERIFIED"
"To ensure stability during a high-speed sprint, I recommend the 
'Shared Types' approach. It provides 90% of the safety of a Monolith with 
0% of the architectural coupling. 

I would also implement 'Versioning' for the types library. The Shell 
can stay on v1.0.0 while the Angular team experiments with v2.0.0. 
Once the Angular team is ready, the Shell lead reviews the 'Breaking 
Changes' log and performs a controlled update. This is called 
'Consumer-Driven Contract' testing."

4. TOOLS TO IMPLEMENT IMMEDIATELY
- HUSKY + LINT-STAGED: To prevent 'junk' code from being committed.
- PLAYWRIGHT: For fast, headless integration testing in CI.
- NPM WORKSPACES (if using a Monorepo): To easily share @org/types 
  without publishing to a registry.
