Redux Toolkit is the official recommendation library for Redux usage in React app as it removes a lot of boilerplate code.

    -> configureStore 
        - instead of createStore (had to mention applyMiddleware separately for redux-thunk/redux-saga). 
        - Redux Dev tools are supported by default and redux-thunk is also included.
        
    -> createSlice 
        - instead of separate actions, action creators, reducers 
        - uses Immer to make state updates safer even if we write mutate code and converts to immutable e.g. state.username = 'abc' 
    -> useSelector, useDispatch - hooks imported from react-redux at component level. It follows Observer pattern.
    -> createAsyncThunk or RTK Query (similar to Tanstack query) for middleware 

The Context Flaw: Context does not have a "selector" mechanism. It is "all or nothing." If the object in the Provider changes, every useContext hook listening to that provider triggers a re-render.

const [userData, setUserData] = useState({ username: 'ABC', age: 20 });
// some state update function
const updateUser = () => {
    setUserData(prev => ({...prev, age: 25}))
}
<Provider userData={userData}>
    <CompA />
    <CompB />
    <CompC />
</Provider>
// Only CompB calls useContext()
Case 1: All 3 components are normal functional components - when provider updateUser() is called, all 3 gets re-rendered. 
Case 2: All 3 are wrapped with React.memo - when provider updateUser() is called, only CompB gets re-rendered. 
The main flaw is if CompB wants to rerender only when username changes, there's no way to check specific object keys. Even though age is updated, it still re-renders. 
Redux uses useSelector so we can specifically mention the object key it wants to listen for changes.

useSelector(state => state.userData.username) // not re-rendered when age changes

----

Redux manages client state
TanStack Query and RTK Query manage server state with caching and synchronization. 
asyncThunk is just a way to perform async logic in Redux, but modern apps prefer dedicated server-state libraries.

Scenarios: 
1) App uses Redux toolkit already with no heavy API call - createAsyncThunk 
2) App doesn't use Redux but handles heavy frequent API call - Tanstack query 
3) App uses Redux and handles heavy frequent API call - RTK Query 

For SSR, createAsyncThunk needs a lot of boilerplate code. Tanstack query and RTK Query are better options.
