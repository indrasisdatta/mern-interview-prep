SOLID principles in React JS:

1. Single Responsibility: A component should have 1 reason to change. instead of having 1 large component, create separate components for different features.
   Eg. Products listing -> Search, Grid, Main (to include both)
   <ProductSearch />
   <ProductGrid />
   
2. Open closed: classes should be open for extension but closed for modification. Component should be extensible using props and composition. 
   Eg. Button component - should have the ability to add icons etc using children.
   <Button> <i class="fa fa-lock"> Disable </Button>
   Each time something is added, we shouldn't have to modify Button
    
3. Liskcov's substitution: objects of superclass should be replaceable with objects of subclass. 
   Eg. Form component with different input fields textarea, input, checkbox. 
   Each field component should implement the same contract value and onChange.
   
4. Interface segregation: should not be forced to depend on interface methods that are not used. Break into multiple interfaces instead.
A component shouldn't depend on prop it doesn't use.
   Eg. create separate components specific to functionality.   
   Good -> <Avatar src={user.avatarUrl} /> 
   Bad  -> <Avatar image={user.imgUrl} name={user.name} onClick={...} />

5. Dependency inversion: higher level modules should not depend on lower level modules, both should depend on abstractions. 
   Classes depends on interfaces or abstract classes rather than concrete implementations. 
   Eg. Instead of directly writing fetch/axios API logic in componen, create separate Service class or function.

DRY Principles:

   Components: If you use the same Button style in 5 places, create a <Button /> component.

   Logic (Custom Hooks): If three different components need to check if a user is logged in, move that logic into a useAuth hook.

   Utilities: Move data formatting (like formatCurrency) into a central utils folder instead of re-writing the math in every component.

   Services: As you mentioned, a separate Service layer ensures that if an API endpoint changes, you only update it in one file, not every component that calls it.

SOLID Principles in NodeJS:

1. Single Responsibility: app.post() shouldn't have all logic in one place. Split it into validator, service, controller 

2. Open/Close: Use the Strategy pattern for adding Google auth in an existing local auth flow. Plug in a new strategy without touching the login class code. 

3. Liskov's substitution: Database class has 2 sub-classes MongoDB and PostgreSQL. Follow the same contract - connect() using same params and return type 

4. Interface Segregation: Don't force a class or function to depend on methods it doesn't use.
Instead of having a single UserService with 50 different methods, split it into smaller interfaces. 

5. Dependency Inversion: UserService directly imports Sendgrid to send email. 
Have a separate EmailService instead and inject it.
