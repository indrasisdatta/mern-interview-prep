Micro-Frontend: 
- App is split into independent modules and deployed separately 
  (e.g. Search, Booking, Payment made in different technologies)
- Host application loads the layout, navbar, routing 
- Remote applications (SearchApp, BookingApp) are exposed via bundles via Module Federation
- Host dynamically loads remote apps at runtime based on the URL 

MF Architecture patterns:
1) Build time integration - child app is published as npm packege and host app imports it (similar to any other dependency).
    // remote app does npm publish 
    export const CheckoutButton = () => {
    return <button>Pay Now</button>;
    };

    // host app imports
    import { CheckoutButton } from '@my-org/checkout';

2) Run time integration 
    - loads MF bundles dynamically at runtime
    - uses Webpack Module Federation

3) Server-side composition 
    - the shell server fetches HTML fragments from MFs 
    - assembles into a single response 
    - sends it to the browser.

4) Edge-side composition
    - the edge CDN (near the user loation) fetches HTML fragments from MFs 
    - assembles into a single response 
    - sends it to the browser.

MICRO FRONTEND (MODULE FEDERATION) – LIMITATIONS & REMEDIES

1) Shared State Management

    Problem:
        Global Redux across micro frontends is hard to manage
        Causes tight coupling between apps

    Remedies: Keep state local to each micro frontend
    Communicate using:
        Custom browser events
        RxJS event bus
        URL state or backend as source of truth

2) CSS Conflicts

    Problem: Styles can leak between micro frontends

    Remedies:
        CSS Modules
        Scoped CSS / BEM
        Shadow DOM (when suitable)
        Shared design system or design tokens

3) CI/CD Complexity

    Problem:
        Each micro frontend needs its own pipeline
        Deployment coordination becomes complex

    Remedies:
        Independent CI/CD per micro frontend
        Backward-compatible contracts
        Clear ownership per team

4) Initial Load Performance

    Problem:
        Multiple remote entry files
        Network request waterfall

    Remedies:
        Lazy load micro frontends
        Preload critical remotes
        Use CDN caching
        Share common dependencies carefully

5) Version Compatibility

    Problem:
        React or shared library version mismatch
        Breaking changes at runtime

    Remedies:
        Strict versioning rules
        Semantic versioning
        Minimize shared dependencies

6) Debugging Complexity

    Problem:
        Errors span across multiple apps
        Harder stack traces and source maps

    Remedies:
        Enable source maps
        Centralized logging and monitoring
        Error boundaries per micro frontend

7) Runtime Failures

    Problem:
        Remote app unavailable
        Bad deployment breaks host at runtime

    Remedies:
        Graceful fallbacks
        Feature flags
        Timeout and retry logic

8) Over-Engineering Risk
    Problem: Too much complexity for small teams or apps

    Remedy: Use micro frontends only when:
        Multiple teams are involved
        Independent deployments are required
        Product is large and long-lived

SUMMARY:
Micro frontends scale teams and deployments but increase architectural
and operational complexity. Use them only when scale demands it.

What if a remote MF goes down? 
 - Error boundaries for each MF to handle runtime errors
    <ErrorBoundary fallback={<Fallback />}>
        <RemoteMF />
    </ErrorBoundary>

 - Dynamic import handles load failure - handles unreachable remoteEntry.js
    import("checkoutApp/Checkout").catch(() => showFallback());

 - Version pinning 
    - Every time a remote app is built, it's uploaded to CDN path: /checkout/v1.3.3/remoteEntry.js
    - Host uses env var CHECKOUT_MF_URL = /checkout/v1.3.3/remoteEntry.js
    - Host uses env var to detect remote URL
        // webpack.config.js (host)
        remotes: {
            checkoutApp: `checkoutApp@${process.env.CHECKOUT_MF_URL}`
        }
    - Once tested, host env can be updated to new version. In case of errors, rollback to previous version.

Q1. How to handle child app A in React 16 and host and child app B in React 18?
Ans. Use ModuleFederation maxVersions or singleton: false for isolation 
     Use web components to sandbox the legacy VDOM 

Q2. How does App A communicate with App B without state pollution? 
Ans. i) Lift state up to host and expose as shared services  
    ii) Use CustomEvent bus or RxJS for decoupled communication, ensuring MF don't directly mutate each other's state 
    - Custom event is fire and forget (if B loads 2 secs late, the message is lost)
     - RxJS uses Observables BehaviorSubject (for 1 message), ReplaySubject (for multiple) so that messages aren't lost 

    // Custom event in App A - update the count 
    const cartEvent = new CustomEvent('UPDATE_CART', { detail: { qty: 1, price: 100 } });
    window.dispatchEvent(cartEvent);
    // Custom event in App B - show updated count
    window.addEventListender('UPDATE_CART', (e) => setCount(e.detail.count));

Q3. Preventing style collissions across different MF 
Ans. i) Use CSS modules or Shadow DOM 
    ii) Shared Design System using CSS vars and common components 
        Expose it as a shared singleton via Module Federation so that browser fetches it only once 
    iii) Tailwind prefixes (each remote should use a unique Tailwind prefix to prevent style collisions.)
    // tailwind.config.js
    module.exports = {
        prefix: 'host-',
    }
    <div class="ra-flex ra-p-4">

Q4. Your app has Checkout, Profile, and Search owned by different teams. How would you split them?
A: Each domain would be a standalone micro-frontend owning its routing, state, and business logic.
The host (shell) app would manage layout and authentication and dynamically load each micro-frontend using Module Federation.
Instead of exposing individual components, each micro-frontend would expose a root entry module (for example CheckoutApp) to keep boundaries clean and reduce coupling.
This allows teams to develop and deploy independently.

Q5: Two micro-frontends use different React versions. How would you handle shared dependencies?
In production systems, React should be shared as a singleton dependency across all micro-frontends with version constraints.
All teams align on compatible React versions to avoid loading multiple runtimes, which can break hooks, context, and increase bundle size.
If different versions are unavoidable, the apps should be isolated (for example via iframe) instead of sharing runtime dependencies.

Q6. A shared Header component is used across 5 apps. How would you design this?
If the Header changes frequently or contains runtime logic, it can be exposed as a shared micro-frontend and consumed via Module Federation.
If it is mostly a stable UI component, a shared component library (design system) is usually better to avoid runtime coupling and reduce failure points.

Q7. One micro-frontend is slowing initial load. How would you optimize?
- Load Micro frontend at route level using dynamic imports and Module Federation, so only required domains load initially 
- Split large remotes into smaller chunks 
- Use prefetching during idle time 
- Avoid sharing heavy unnecessary dependencies 

Q8. How do teams deploy independently without breaking others?
- Each MF to have it's own CI/CD pipeline and deployment cycle 
- Host app to have graceful fallback UI if remote app fails to load 
To avoid runtime breakage:
- Exposed modules should follow semantic versioning 
- Changes are backwards compatible 
- Runtime version pinning/feature flags can be used 

Q9. How do you avoid loading duplicate dependencies?
Common libs like React, ReactDOM are configured as shared singleton dependencies in Module Federation with strict version requirements. 

Only common UI - use shared library/design system 
Business logic + UI - use micro frontend app 
npm publish follows semantic versioning 
v1.0.0  → initial release
v1.0.1  → bug fix
v1.1.0  → backward-compatible feature
v2.0.0  → breaking change

Remote will NOT detect the host app prop change automatically.
Detection happens only if:
 - runtime breaks
 - or shared types fail
 - or contract tests exist

Tools like Pact, contact test 

Remedies:
• At build time, maintain shared types in @common/contracts and ensure backward compatibility.
• Run TypeScript checks in the remote CI (tsc --noEmit) to catch contract drift before deployment.
• At runtime, apply Zod validation at micro-frontend boundaries to guard against unexpected shape changes.
