Upgrading React in a large codebase can be tricky. Here’s how I did it safely:

1️⃣ Start fresh

Create a new branch

Delete node_modules & package-lock.json

2️⃣ Upgrade core packages

npm install react@latest react-dom@latest
npm install --save-dev @types/react@latest @types/react-dom@latest


3️⃣ Update minor dependencies

npm update
npm outdated
npm install


⚠️ ERESOLVE errors? Upgrade specific packages with --legacy-peer-deps

4️⃣ Automate refactors with codemods

npx codemod@latest react/19/migration-recipe

Handles forwardRef, lifecycle renames, etc.

5️⃣ Upgrade supporting libraries

React Query v3 → @tanstack/react-query v5

Update queryKey to [] notation & queryFn

npm uninstall react-query --legacy-peer-deps
npm install @tanstack/react-query --legacy-peer-deps

--------------

Note: Always upgrade in a separate branch, run codemods first, fix breaking tests, then gradually roll out.

npm install fails if it finds conflicting peer dependencies. For e.g. some-lib expects React 18 but you're on React 19.

--legacy-peer-deps is a shortcut, not a fix. You can get your app running, but for stable production, all 3rd-party deps and types should eventually support React 19.

Normally, npm enforces peer dependency compatibility.

With --legacy-peer-deps, npm ignores peer dependency conflicts and installs the packages anyway.
    some-lib expects React 18
    You're on React 19 → npm will install it anyway
    Npm won't complain about peer conflicts
    But runtime or TS issues may occur if the library truly relies on older APIs

--force
  Forces npm to install packages regardless of conflicts or errors.
  It overrides everything, including version mismatches, broken engines, or existing locks.
  Risky: can break your node_modules, produce subtle runtime bugs, or cause weird CI/CD failures.

npm install does these checks:
1) Version match - packages are satisfying the version. 
    "react": "^19.0.0" → npm makes sure it picks a compatible version like 19.2
2) Peer dependencies - You have react19 but a package uses react18, so it will show warning
3) Node and npm version engine check - some packages work only on specific node, npm versions 
4) Integrity/corruption checks - ensures downloaded packages aren't corrupted by matching checksum
5) Lifecycle scripts - runs any setup script that the package requires preinstall, postinstall 

--legacy-peer-deps ignores only 2 but does the other checks
--force ignores 1, 2, 3
