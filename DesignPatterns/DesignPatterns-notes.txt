Common Frontend design patterns:

1) Singleton: Ensure that a class/object has only one instance and provides a global point of access to it.
Drawbacks: 
- hard to test shared instance (need to reset manually before each test)
- Singleton lives outside React's lifecyle so doesn't re-render when data is changed. 
Use case: When UI update is not needed e.g. API Client (Axios instance)
Module-level singleton.
In ESM module system, the first time a file is imported, the results are cached. Every subsequent import returns the exact same memory reference. 
NodeJS use case: DB connection, config, logger

2) Factory: instead of using new keyword or manually defining which component to load in every file, use a Factory component. 
- Decoupling 
- Scalability (add the Factory file only for future addition)
Also satisfies Liskov's substitution O-C compliant 
NodeJS use case: NotificationFactory for email and SMS 

3) Observer: many listeners react to a change in one source. 
Subject holds the state/data 
Observer gets notified when data changes 
In React, State/store is the subject and components are observers.
Libraries like RxJS or react-observable are also valid examples. 
NodeJS use case: Events, queues, notifications. EventEmitter uses it.

4) Strategy: enables selecting an algorithm at runtime. Instead of big if/else or switch, encapsulate each behavior in its own function and choose dynamically. 
NodeJS use case: Passsport.js uses strategies for Google, FB and local auth. We plug in the strategy we need and the main app flow remains unchanged.

5) Adapter - transform incoming data from API layer to a structure that components can use effectively 
- Adapter pattern is implemented as a lightweight mapping function alongside API services like Axios 
- For small to medium apps, Adapters often live in the same service file 
NodeJS use case: Libraries like Prisma acts as adapters allowing us to write standard JS that gets adapted to specific SQL dialects for PostgreSQL, MySQL, SQLite

6) Facade - provides a simple interface over complex logic
- composes multiple custom hooks and business logic  
NodeJS use case: BFF acts as a facade by collecting data from 5 different microservices and returning one clen JSON response.  

7) Decorator - add behavior without modifying original function/component. 
E.g. HOC, function wrappers (logging, permissions, styling)
NodeJS use case: middleware

8) Proxy - control access or add logic before real object - interact with the Proxy object instead of the main object directly
E.g. Axios interceptor (proxy like behavior)
Cache, validation layer 
NodeJS use case: control access, caching, rate limiting

React Design Patterns:

1) Hooks: 
 - enable logic reuse through functional composition 
 - Improves readabiity (no nested wrapper like HOC)
 - works natually with Suspense and use()
Composition example - 3 hooks useAuth, useTheme, useData used in a single component (3 level nested HOC becomes complex and props may conflict)
In React 19, hooks are useful 90% of the time.
HOC - useful for cases like Error Boundaries, Auth Guards, Feature Flags. 

2) Compound components: Break a large component into multiple sub-components that are closely related. 
E.g. Modal - ModalHeader, ModalBody, ModalFooter are related to the Modal but they don't exist independently.
Use case: Tabs, Accordion, Modal (In Material UI)
Note: 
- Sub-components should be related to the main component 
- Don't import sub-component and try to implement indepedently 

<Tabs>
  <Tabs.List />
  <Tabs.Panel />
</Tabs>

Parent (Tabs)
 - Creates the context
 - Holds shared state (active tab, handlers, etc.)
 - Wraps children with <TabsContext.Provider>

Sub-components (Tabs.List, Tabs.Panel, etc.)
 - Call useContext(TabsContext)
 - Read/update shared state
 
3) Container/Presentational:
 - Container component (logic): Fetches data, Handles state, Business logic
 - Presentational component (UI): Receives props, Just renders UI, No business logic
Modern approach: custom hooks + UI component 
Use cases: User list/table/card, Dashboard (charts UI + data logic)

Old: UserContainer.jsx, UserList.jsx 
Modern approach for small components: useUsersData.jsx, UserList.jsx (call custom hook in this file)

4) HOC pattern: 
HOC function - takes a function as argument and returns a function (map, filter, reduce)
HOC issue in legacy React code - JS doesn't inherit static properties automatically.
class UserPage extends React.Component {
  static fetchData() {
    // used for SSR or routing
  }
}
export default withAuth(UserPage);

UserPage.fetchData ❌ undefined

Fix using: 
import hoistNonReactStatics from "hoist-non-react-statics";
hoistNonReactStatics(HOC, WrappedComponent);

5) Render props: value of prop expects a function which returns JSX 
Drawbacks: 
- nested structure makes JSX messy
- new function created in every render 
Use case: Form libaries like Formik (old version)

6) Slot pattern: if there's no shared states between child components, then instead of using compound components, we can use slot. 
<Card 
  header={<h4>Card Heading</h4>}
  content={<p>Card content displaying descriptionb</p>}
  footer={<p>Card footer</p>}
/>
